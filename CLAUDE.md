# Working in this repo

You are an LLM editing a [TiddlyWiki](https://tiddlywiki.com)-backed knowledge base. Content lives in `vault/` as Markdown files with YAML frontmatter. The human reads it in a browser. A live filesystem watcher syncs your edits to the browser without a reload.

## What you write to

- **`vault/`** — write all content here. One tiddler per file.
- **`vault/.obsidian/`** — leave alone. Obsidian's config; no tiddlers live here.

## What you don't touch

- **`wiki/tiddlers/`** — system tiddlers and the dynamic-store loader. The user owns this.
- **`wiki/output/`** — generated by `npm run build`. Never hand-edit.
- **`tiddlywiki.info`** — wiki configuration. Don't modify unless asked.

If a save accidentally lands in `wiki/tiddlers/` (e.g. `$__StoryList.tid`), delete it.

## Tiddler format

Markdown with YAML frontmatter:

```markdown
---
title: Some Concept
tags: [concept, synthesis]
rating: 7
---

Body in Markdown. Wiki links use [[Target Title]] or [[Display Text|Target Title]].
```

Filename is `vault/<Title>.md`. The frontmatter `title` wins over the filename when they differ — prefer keeping them in sync.

### Field conventions

- **`title`** — always include. Wins over filename.
- **`tags`** — YAML array. Multi-word tags are fine: `[concept, multi word tag]`. Tags become TiddlyWiki bracketed lists on disk; round-trips correctly.
- **`rating`** — integer 1–10 (optional). Indicates importance/quality.
- **Custom fields** — fine to invent (e.g. `author`, `year`, `status`). They become queryable tiddler fields.
- **Don't include** `created`, `modified`, `type` — TiddlyWiki manages timestamps; `.md` extension implies the type. The serializer will round-trip these out anyway.

## Linking

- `[[Target Title]]` — link by exact tiddler title
- `[[Display Text|Target Title]]` — link with custom display text
- Targets that don't exist yet are fine — they show as "missing" links and become normal links when the target is created. Use them as TODO markers for content gaps.

## When to use `.tid` instead of `.md`

Default to `.md` for everything. Use `.tid` only for:
- TiddlyWiki UI/view tiddlers (wikitext that uses `<<macros>>`, `{{transclusions}}`, widget syntax)
- Tiddlers where the body is computed (filter expressions, dashboards)

`.tid` format example:
```
title: My Custom View
tags: $:/tags/SideBar
caption: Concepts

<<list-links "[tag[concept]sort[title]]">>
```

`.tid` files go in `vault/` too.

## Workflow

- **Add a tiddler** — write a new `.md` file in `vault/`. The watcher picks it up within ~400ms.
- **Edit a tiddler** — edit the file in place. Same.
- **Rename a tiddler** — rename the file. The frontmatter `title` is what the wiki uses; rename either or both.
- **Delete a tiddler** — delete the file.
- **Add a link before the target exists** — fine. Create the target later when you have content for it.

## Style

- Tiddler bodies should be concise, fact-dense, and link-rich. Prefer linking to a related tiddler over restating its content.
- Wiki links are cheap; create stub tiddlers for important referenced concepts.
- Don't add `## See also` sections out of habit — only when there are non-obvious related tiddlers worth pointing to.
- Don't pad. Each section should earn its place.

## Things to know

- The browser autoreloads via polling (`$:/config/SyncPollingInterval` is set to 2 seconds in this repo).
- Filesystem watching means changes from any source (you, the human, `git pull`) are picked up live.
- The wiki lives at `http://localhost:8080` when `npm start` is running. The human controls that process.

## Things not to assume

- The wiki may not be running when you make changes. That's fine — your edits are the source of truth and will be loaded next start.
- The human may have edited tiddlers in the browser. Read the file before editing to see current state.
- Don't run `npm start` or `npm run build` unless asked.
